Implementation: Define a Performance Metric¶

It is difficult to measure the quality of a given model without quantifying its performance over training and testing. This is typically done using some type of performance metric, whether it is through calculating some type of error, the goodness of fit, or some other useful measurement. For this project, you will be calculating the coefficient of determination, R2, to quantify your model's performance. The coefficient of determination for a model is a useful statistic in regression analysis, as it often describes how "good" that model is at making predictions.

The values for R2 range from 0 to 1, which captures the percentage of squared correlation between the predicted and actual values of the target variable. A model with an R2 of 0 always fails to predict the target variable, whereas a model with an R2 of 1 perfectly predicts the target variable. Any value between 0 and 1 indicates what percentage of the target variable, using this model, can be explained by the features. A model can be given a negative R2 as well, which indicates that the model is no better than one that naively predicts the mean of the target variable.

from sklearn.metrics import r2_score

def performance_metric(y_true, y_predict):
    """ Calculates and returns the performance score between 
        true and predicted values based on the metric chosen. """
    
    score = r2_score(y_true, y_predict)
    
    return score

Question 3 - Training and Testing¶

What is the benefit to splitting a dataset into some ratio of training and testing subsets for a learning algorithm?
Hint: What could go wrong with not having a way to test your model?

Answer: Splitting a dataset into training and testing subsets allows one to establish a performance measure which can be extrapolated to out of sample data with unknown labels or target values. In other words, the score that the model arrives at on the testing data is the best estimate of how the model will perform one real world data that one can obtain using machine learning techniques.

To elaborate, a researcher can use the training data to prepare an algorithm and then deploy it on the testing data subset. He or she can then evaluate the performance of the model against a certain metric and also whether or not it suffers from too much bias or variance. If the model's performance is unsatisfactory, then optimization techniques can be used, hyperparameters of the model can be tuned, or a new algorithm entirely can be chosen to learn and predict patterns in the data. The benefit of having test data is that it gives a consequence-free sandbox-like environment in which a model can be observed, tested, and tweaked. If a weak machine learning algorithm is prematurely deployed on real data, nasty consequences could emerge. For example, in financial markets it could result in the loss of one's trading capital. In medicine, it could result in patients with certain diseases being misdiagnosed. Without having a way to test a model, it could contain a great deal of bias or variance and one wouldn't even know it until it is too late.

Question 7 - Grid Search¶

What is the grid search technique and how it can be applied to optimize a learning algorithm?

Answer: Grid search is a brute-force technique used to optimize the hyperparameters of a learning algorithm. Using this technique, a researcher can directly input several dictionaries of hyperparameter lists that they would like to investigate. These could include the max depth of a decision tree; the kernel, C, or gamma in an SVM; or the number of neighbors in KNN. A grid of all possible combinations of given parameters is computed and then each set is tested on the training data one by one. The grid search technique will then return the algorithm that best fits the training data, as well as its corresponding hyperparameters, and score. Once the optimal model is found, it can be directly tested on the final test data for a final evaluation. In the case that performance is still weak, perhaps a new grid search needs to be performed on sets of different parameters. It's even possible that another algorithm could better model the dataset. Maybe there were some mistakes in how the data was collected or processed. This goes back to the importance of having a test set that is separate from the training set. Grid search is useful but not a solution to all the problems that one could encounter.

Question 8 - Cross-Validation¶

What is the k-fold cross-validation training technique? What benefit does this technique provide for grid search when optimizing a model?
Hint: Much like the reasoning behind having a testing set, what could go wrong with using grid search without a cross-validated set?

Answer: During the training process of developing a learning algorithm, employing KFold cross-validation divides a set of training data into k equal partitions, each of which are used as a validation subset exactly once. The rest of the data that is not partitioned is used as the training subset that the model learns from initially. A model using KFolds runs k cycles over the data, setting aside a section for validation, training on the rest, then making predictions on the labels or target values of the validation subset. In each cycle, the predicted values, gleaned from the training process, are compared against the known values. A score is generated for each cycle, and finally all the scores are averaged together to give a more valid metric of the predictive power of the model. By performing all of the above, KFold cross-validation is much less subject to possible bias that could emerge from clusters that are present in the dataset. For example, it's not uncommmon for one class label to be very prominent in the first half of the dataset, and not present in the second. Merely splitting the data into training and testing sets of equal size would cause the algorithm to learn patterns in the training set that are largely irrevelant to the testing set. Performance would be poor, even though the model itself could otherwise be robust if a different split was used. Setting shuffle=True in a KFold object goes even further to ensure the data is set in a random order before cross-validation is performed so that this error does not occur.

In regards to grid search, using KFolds ensures that the sets of parameters that the researcher supplies have a chance to be validated against multiple folds of the training data. If maximum accuracy is sought, then the more folds the better. This prevents the grid search from selecting parameters that it determines are optimal, albeit for only the training data as a whole. Grid search without KFolds would indeed provide optimal parameters for the very data that it has just trained on, however these may not be optimal for data outside of the training set, e.g. testing data or out of sample data. Using KFolds during the training process can reduce this possibility to some degree.

from sklearn.tree import DecisionTreeRegressor
from sklearn.metrics import make_scorer, r2_score
from sklearn.grid_search import GridSearchCV

def fit_model(X, y):
    """ Performs grid search over the 'max_depth' parameter for a 
        decision tree regressor trained on the input data [X, y]. """
    
    # Create cross-validation sets from the training data
    cv_sets = ShuffleSplit(X.shape[0], n_iter = 10, test_size = 0.20, random_state = 0)

    # TODO: Create a decision tree regressor object
    regressor = DecisionTreeRegressor()

    # TODO: Create a dictionary for the parameter 'max_depth' with a range from 1 to 10
    params = {'max_depth':list(range(1,11))}

    # TODO: Transform 'performance_metric' into a scoring function using 'make_scorer' 
    scoring_fnc = make_scorer(performance_metric)

    # TODO: Create the grid search object
    grid = GridSearchCV(regressor,params,scoring=scoring_fnc,cv=cv_sets)

    # Fit the grid search object to the data to compute the optimal model
    grid = grid.fit(X, y)
    
    # Return the optimal model after fitting the data
    return grid.best_estimator_

No matter how the data is sliced and diced, predictions that the model generates don't drift too far from each other. An R^2 of 0.78 is definitely notable, but it's not groundbreaking in my opinion. A range of 69,044.61 after 10 splitting trials is considerable on the surface, but actually not such a big deal. What would be more useful to consider is the mean and standard deviation in price predicitions that emerge from 1000 random splits. Homes sell for 5 to 10% below or in excess of their asking price all the time. When using any model to inform real world decisions, its imporant to stay flexible and keep an open mind. I wouldn't use this model as my only tool in assessing a home's worth, but could be useful in conjuntion with other tools though.

In it's current form, the data collected in Boston would not be too applicable to predicting home values in a rural city. Living in urban cities has always come with a higher price premium over rural areas. This in itself will ensure that any given urban home is almost certainly more expensive on a raw basis than a rural home, all else held equal. What would make the data more applicable though is standardization. If the data was standardized, the urban price premium would be controlled for, and a more relative basis could be establish for carrying patterns learned in the home values of urban areas to those in rural areas. It is likely that tastes and preferences differ between the two areas that would still present some problems however. Once again, further investigation is needed.
